# План реализации проекта "Orders & Fulfillment"

## 1. Структура проекта

Проект будет организован согласно принципам чистой архитектуры (Clean Architecture) с разделением на слои:

```
.
├── api/
│   └── proto/               # Определение gRPC сервисов (.proto файлы)
├── cmd/
│   └── app/
│       └── main.go          # Точка входа в приложение
├── internal/
│   ├── config/              # Загрузка конфигурации (.env, env vars)
│   ├── domain/              # Доменные модели (Order, Item, Status)
│   ├── repository/
│   │   └── postgres/        # Реализация работы с БД через pgx
│   ├── service/             # Бизнес-логика (OrderService, FulfillmentService)
│   └── transport/
│       ├── http/            # Echo хендлеры и роутинг
│       └── grpc/            # gRPC сервер и клиент
├── migrations/              # SQL миграции (goose)
├── docker-compose.yml       # Оркестрация (App + Postgres)
├── Dockerfile               # Сборка приложения
└── go.mod
```

## 2. Пошаговый план реализации

### Этап 1: Подготовка и Доменная модель
1.  Определить структуру `Order` и `OrderItem` в `internal/domain/order.go`.
2.  Определить gRPC контракт в `api/proto/fulfillment.proto` для обработки заказов.
3.  Сгенерировать Go-код из proto-файла.

### Этап 2: Слой работы с данными (Repository)
1.  Реализовать методы в `internal/repository/postrgres/db.go`:
    *   `CreateOrder`: сохранение нового заказа.
    *   `GetOrder`: получение заказа по ID.
    *   `ListOrders`: фильтрация заказов по `customer_id` и `status`.
    *   `UpdateOrderStatus`: обновление статуса заказа.

### Этап 3: gRPC Fulfillment Service
1.  Реализовать gRPC сервер в `internal/transport/grpc/`:
    *   Принимает запрос на обработку.
    *   Меняет статус на `PROCESSING`.
    *   Имитирует работу (sleep).
    *   Меняет статус на `FULFILLED` или `FAILED`.
2.  Реализовать gRPC клиент для вызова этого сервиса из основного приложения.

### Этап 4: Бизнес-логика (Order Service)
1.  Реализовать `OrderService` в `internal/service/order_service.go`:
    *   Координация создания заказа: сохранение в БД -> запуск асинхронной gRPC обработки.
    *   Получение данных о заказах.

### Этап 5: HTTP API (Echo)
1.  Настроить Echo в `internal/transport/http/`:
    *   `POST /orders`: создание заказа.
    *   `GET /orders/:id`: получение заказа.
    *   `GET /orders`: список заказов.
2.  Добавить middleware для логирования.

### Этап 6: Сборка и Docker
1.  Написать `Dockerfile` для Go-приложения.
2.  Создать `docker-compose.yml`:
    *   Сервис `db` (Postgres).
    *   Сервис `app` (зависит от `db`).
    *   Настройка переменных окружения для связи компонентов.

### Этап 7: Документация и Тестирование
1.  Подготовить `README.md` с инструкциями.
2.  Проверить работоспособность через `curl` или `Postman`.

## 3. Технические решения
*   **БД:** Используем `pgxpool` для эффективного управления соединениями.
*   **Асинхронность:** При создании заказа HTTP-ответ возвращается сразу после записи в БД, а вызов gRPC выполняется в отдельной горутине (или через очередь, если бы это требовалось, но для ТЗ достаточно горутины).
*   **Конфигурация:** Использование `cleanenv` или стандартного `os.Getenv` для работы с `.env`.
